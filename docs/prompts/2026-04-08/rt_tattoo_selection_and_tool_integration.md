# RT 토론: 문신 선택 전략과 외부 도구 통합 지점

## 배경

제멘토는 상태 전이 시스템이다.
소형 LLM(Gemma 4 E4B, 4.5B 유효 파라미터, 128K 컨텍스트)이
매 루프마다 컨텍스트를 완전히 잃고,
구조화된 외부 상태(문신, Tattoo)만을 입력으로 받아 추론을 이어간다.

이미 확정된 설계:
- 문신은 상태(state)이지 로그가 아니다
- 문신 스키마: goal(불변) + state(phase, assertions, open_questions, next_directive) + integrity
- Assertion은 추가/무효화만 가능, 수정 불가
- Phase: DECOMPOSE → INVESTIGATE → SYNTHESIZE → VERIFY → CONVERGED
- 승격 기준: 관련성 + 근거성 + 비모순성 + 원자성

## 모델 제약 조건 (Gemma 4 E4B)

- 컨텍스트 윈도우: 128K 토큰
- 하이브리드 어텐션 (512 토큰 sliding window + 전역 어텐션 레이어)
- 유효 파라미터: 4.5B
- 내장 추론 모드 (step-by-step thinking)
- 네이티브 function-calling 지원
- 소형 모델의 특성: 긴 입력에서 중간부 정보 손실 경향 (lost-in-the-middle)

## 토론 주제 1: 문신 선택 (Tattoo Selection)

### 문제

루프가 반복되면 assertion이 누적된다.
assertion이 많아지면:
1. 문신이 컨텍스트 윈도우의 대부분을 차지하여 추론 공간이 줄어든다
2. 소형 모델은 긴 입력의 중간부를 놓치는 경향이 있다
3. 불필요한 assertion이 노이즈가 되어 추론 품질이 저하된다

### 결정해야 할 것

1. **Assertion 상한**: 한 번에 몇 개까지 포함할 것인가?
   - 고정 상한 vs 동적 상한(phase에 따라 다르게)?

2. **선택 전략**: 상한을 넘으면 어떤 assertion을 포함할 것인가?
   - 선택지:
     - (A) confidence 기반: 높은 confidence 우선
     - (B) recency 기반: 최근 루프에서 생성된 것 우선
     - (C) goal-relevance 기반: 현재 next_directive와 관련도 높은 것 우선
     - (D) 복합: 위 기준의 가중 조합
     - (E) 계층화: 핵심 assertion은 항상 포함, 나머지를 선택
   - 선택 자체를 LLM이 하는가, 규칙 기반으로 하는가?

3. **압축 가능성**: 여러 assertion을 하나로 요약할 수 있는가?
   - 요약하면 정보 손실 위험
   - 요약 안 하면 공간 부족
   - 요약 자체가 오류 주입 지점이 될 수 있다

4. **배치 전략**: 소형 모델의 lost-in-the-middle 특성을 고려할 때,
   assertion을 문신 내에서 어떤 순서로 배치해야 하는가?
   - 중요한 것을 앞/뒤에 배치?
   - goal → high-confidence assertions → next_directive 순서가 적절한가?

### 제약

- 선택 전략 자체가 오류 전파의 새로운 경로가 되면 안 된다
- 선택에서 제외된 assertion은 사라지는 것이 아니라 저장소에 남아 있어야 한다
- 선택 로직은 LLM 추론 없이도 실행 가능해야 한다 (오케스트레이터 레벨)

## 토론 주제 2: 외부 도구 통합 지점 (Tool Integration Point)

### 문제

현재 설계에서 외부 도구/정보는 루프 내 optional 단계다:

```
1. 컨텍스트 없음
2. 문신 선택
3. 외부 정보/툴 (optional)  ← 여기
4. 추론 수행
5. 새 문신 생성
6. 상태 업데이트
```

Gemma 4 E4B는 네이티브 function-calling을 지원한다.
이것이 설계에 어떤 영향을 주는가?

### 결정해야 할 것

1. **도구 호출의 주체**: 누가 도구를 호출하는가?
   - (A) LLM이 직접 function-calling으로 호출 (Gemma의 네이티브 기능 활용)
   - (B) 오케스트레이터가 문신의 next_directive를 해석하여 호출
   - (C) 하이브리드: LLM이 요청하고, 오케스트레이터가 실행하고, 결과를 LLM에게 돌려줌

2. **도구 결과의 신뢰도**: 도구 결과는 어떤 confidence로 assertion에 반영하는가?
   - 외부 사실(검색 결과, 계산 결과)은 LLM 추론보다 신뢰도가 높은가?
   - 현재 설계: "외부 도구 없이 도출된 주장은 confidence 상한 0.7"
   - 도구 결과 기반 assertion은 상한을 해제하는가? 해제한다면 조건은?

3. **도구 호출과 루프의 관계**: 도구 호출이 루프를 소비하는가?
   - (A) 도구 호출은 루프 내부에서 발생 (같은 루프에서 호출+추론)
   - (B) 도구 호출은 별도 루프를 소비 (호출 루프 → 결과 해석 루프)
   - 소형 모델의 제한된 추론 능력을 고려하면, 한 루프에서 도구 호출 + 결과 해석 + 추론을 다 하는 것이 품질에 어떤 영향을 주는가?

4. **도구 오류 처리**: 도구가 잘못된 결과를 반환하면?
   - 도구 결과도 승격 기준(관련성, 비모순성)을 통과해야 하는가?
   - 도구 오류가 높은 confidence로 assertion에 들어가면 오류 증폭이 가속된다

### 제약

- 도구 통합이 상태 전이의 결정론적 성격을 훼손하면 안 된다
- 도구 호출 실패가 루프 체인 전체를 중단시키면 안 된다
- 도구 의존도가 높아지면 제멘토의 본질(문신 기반 상태 전이)이 희석될 수 있다

## 토론 규칙

1. 각 참여자는 **주제별로 입장을 명시**하라 (선택지 번호 + 근거)
2. 다른 참여자의 의견에 **구체적으로 반론하거나 보강**하라
3. "다 좋다", "상황에 따라 다르다"는 답변 금지. 하나를 골라 근거를 대라
4. 소형 모델(4.5B)의 실제 능력 한계를 고려하여 판단하라
5. 이론적 우아함보다 **실제로 작동할 가능성**을 우선시하라
6. 3라운드 내에 수렴하지 못하면 4라운드에서 의장이 결론을 내린다
