# RT 토론: E4B × 3 직렬 구조 (A-B-C) 설계

## 배경

### 실험 결과 요약 (0~3)

제멘토는 Gemma 4 E4B(4.5B)를 사용한 상태 전이 시스템이다. 실험 0~3의 결과:

| 실험 | 결과 | 의미 |
|------|------|------|
| 실험 0 (Baseline) | 단일 추론 정답률 50% | 복잡한 문제는 단일 추론으로 불가 |
| 실험 1 (Assertion Cap) | JSON 성공률 100%, 12개까지 안정 | E4B는 문신을 읽을 수 있다 |
| 실험 2 v1 (모델 자율 phase 전이) | **수렴 0%** | E4B 1개가 추론+구조판단 동시 수행 불가 |
| 실험 2 v2 (Python 강제 phase 전이) | **수렴 94.4%, 정답률 94.4%** | 구조를 잡아주면 추론은 잘 함 |
| 실험 3 (결함 주입) | **오류 감지율 0%** | E4B는 자기 assertion 자가 검증 불가 |

### 핵심 발견

1. **E4B는 유능한 실행기이나 무능한 판단자/검증자**
2. 실험 2 v1에서 실패한 건 "1개 E4B가 모든 걸 한 것"
3. 실험 3에서 실패한 건 "자기 자신의 assertion을 자기가 검증한 것"

### 미검증 가설: A-B-C 직렬 구조

원래 가설은 E4B 3개를 역할 분리하여 직렬로 동작시키는 것이었다:

```
A (제안자): 문신을 읽고 추론, assertion 생성
    ↓ 문신 전달
B (비판자): A의 assertion을 비판/무효화
    ↓ 문신 전달
C (판정자): B의 비판이 수렴했는지 판단 → phase 전이 결정
    ↓ 문신 전달 (next_directive 포함)
A → B → C → ... (반복)
```

**이 구조에서는 Python 오케스트레이터가 불필요하다.** 문신만으로 A→B→C 사이의 상태 전달과 구조 판단이 이루어진다.

실험 2 v1/실험 3과의 결정적 차이:
- B는 A의 assertion을 **교차 검증** (자가 검증이 아님)
- C는 "추론이 맞는가"가 아니라 **"B의 비판이 반복되기 시작했는가"만 판단** (패턴 비교)

## 토론 주제 1: A-B-C 역할 설계

### A (제안자)의 역할

현재 실험 2의 시스템 프롬프트와 유사. 이미 검증됨.
- 문신을 읽고 next_directive를 따른다
- 새 assertion 생성, 질문 해결
- final_answer 제출

### B (비판자)의 역할 — 토론 필요

**핵심 질문: B는 구체적으로 무엇을 출력해야 하는가?**

선택지:
1. **assertion 단위 비판** — 각 assertion에 대해 "유효/의심/무효" 판정 + 이유
2. **전체 추론 비판** — A의 전체 추론 흐름에서 약점/모순/빠진 부분 지적
3. **혼합** — assertion 판정 + 전체 추론 코멘트

고려사항:
- B도 E4B(4.5B)이다 — 너무 복잡한 비판 프롬프트는 JSON 파싱 실패 위험
- B의 출력이 C의 입력이 된다 — C가 "비판 수렴"을 판단하려면 B의 출력 형식이 일관되어야
- 실험 3에서 자가 검증은 실패했지만, 교차 검증은 다를 수 있다

### C (판정자)의 역할 — 토론 필요

**핵심 질문: C는 "비판 수렴"을 어떻게 판단하는가?**

C가 받는 입력:
- 현재 문신 (A의 assertion + B의 비판)
- B의 이번 라운드 비판
- B의 이전 라운드 비판 (문신에 기록되어야 함)

C의 출력:
- phase 전이 여부
- 다음 next_directive

질문:
- B의 비판 히스토리를 문신에 어떻게 저장하는가? (assertion으로? 별도 필드?)
- C의 프롬프트를 "두 텍스트가 비슷한지 비교하라" 수준으로 단순화할 수 있는가?
- C가 실패하면 (잘못된 phase 전이) 어떻게 되는가? 안전장치는?

## 토론 주제 2: 문신 스키마 확장

현재 문신 스키마는 단일 에이전트를 가정한다. A-B-C 구조에서는:

### 필요한 변경

1. **역할 식별자** — 문신에 "이번 턴의 수신자는 A/B/C" 표시 필요?
2. **비판 기록** — B의 비판을 assertion과 별도로 저장? 아니면 assertion에 통합?
3. **비판 히스토리** — C가 수렴을 판단하려면 이전 비판을 볼 수 있어야 함. 몇 라운드치?
4. **phase 전이 기록** — C의 판정 이력도 문신에 남겨야 하는가?

### 스키마 확장 방향

선택지:
- **A안: 최소 확장** — 기존 스키마에 `role_target`과 `critique_log` 필드만 추가
- **B안: 별도 문신** — A/B/C 각각 다른 문신 스키마. 전달 시 변환
- **C안: 공유 문신 + 뷰** — 하나의 문신이지만 역할별로 보이는 필드가 다름

## 토론 주제 3: 순환 구조와 종료 조건

### 정상 흐름

```
A → B → C → A → B → C → ... → 수렴
```

### 질문

1. **한 phase 내에서 A-B-C를 몇 사이클 돌리는가?** 
   - 고정? (예: 최대 3사이클)
   - C가 결정? (비판 수렴 시 다음 phase로)
   - 둘 다? (C가 결정하되, 상한은 고정)

2. **phase 전이 시 B의 비판 기록은 리셋하는가?**
   - 리셋하면: C가 새 phase에서 다시 수렴 판단 가능
   - 유지하면: 맥락이 풍부하지만 문신이 비대해짐

3. **C가 수렴을 오판하면?**
   - 너무 빨리 전이 → 추론 불충분
   - 너무 늦게 전이 → 루프 낭비
   - 안전장치: Python에서 최소/최대 사이클 강제?

4. **Python 오케스트레이터의 잔여 역할**
   - A-B-C 직렬 구조에서도 Python이 해야 하는 것: 호출 순서 관리, 문신 직렬화/역직렬화, 최대 루프 상한
   - 이건 "오케스트레이션"이 아니라 "인프라" — 제멘토 가설에 위배되지 않음

## 제약 조건

- 모델은 Gemma 4 E4B 고정 (4.5B, Q4_K_M)
- A, B, C 모두 같은 모델, 다른 시스템 프롬프트
- 매 호출마다 컨텍스트 완전 초기화 (기억상실)
- 상태 전달은 오직 문신을 통해서만
- 문신 토큰 예산: assertion soft cap 8 / hard cap 10 (기존 RT 결론)
- 비판 기록이 추가되면 총 토큰 예산 재검토 필요

## 각 참여자에게

실험 데이터에 근거하여 답변하라. 추측보다 실험 결과에서 추론하라.

특히:
- 실험 3에서 "자가 검증 0%"였으므로, B의 교차 검증이 왜 다를 수 있는지 구체적 근거를 대라
- 실험 2 v1에서 phase 전이 실패(0%)였으므로, C의 "비판 수렴 판단"이 왜 다른 난이도인지 설명하라
- 4.5B 모델의 한계를 고려하여, 각 역할의 프롬프트가 얼마나 단순해야 하는지 제안하라
